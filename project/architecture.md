# Конструктор печатной формы «Счёт на оплату» для ЗаказКлиента

## 1. Техническое задание

### Исходная задача
Клиенты хотят разные варианты печати счёта на оплату из документа ЗаказКлиента:
- Шапка с полными реквизитами или краткая
- Детальная таблица товаров или только итоги
- Разные варианты подписей

### Функциональные требования
1. Менеджер сам выбирает, какие блоки включить в печатную форму
2. Менеджер задаёт порядок блоков
3. Возможность сохранения набора блоков как пресета (по пользователю)
4. Печать и сохранение в PDF через типовой механизм `УправлениеПечатью` БСП

### Нефункциональные требования
- Минимальные изменения типовой конфигурации УТ 11.5
- Максимальное использование типовых механизмов и программного интерфейса
- Строгое разделение клиент/сервер
- Соответствие стандартам именования 1С

---

## 2. Архитектурное решение

### Подход
Новая обработка печати + форма настройки блоков. Не модифицируем типовую `ПечатьСчетовНаОплату`, а создаём отдельную обработку, которая регистрируется как дополнительная печатная форма для `Документ.ЗаказКлиента`.

### Блоки печатной формы

Каждый блок имеет единый идентификатор, используемый в конструкторе, коде и макете.

| # | Идентификатор блока | Описание | По умолч. | Группа | Области макета |
|---|---|---|---|---|---|
| 1 | `ШапкаПолная` | Полные реквизиты организации (ИНН, КПП, адрес, банк) | Вкл | Шапка | `ШапкаПолная` |
| 2 | `ШапкаКраткая` | Только наименование и ИНН | Выкл | Шапка | `ШапкаКраткая` |
| 3 | `РеквизитыПокупателя` | Данные контрагента/партнёра | Вкл | — | `РеквизитыПокупателя` |
| 4 | `ТаблицаТоваровДетальная` | Полная таблица (номенклатура, кол-во, цена, сумма, НДС) | Вкл | Товары | `ТаблицаТоваровДетальная_Шапка`, `ТаблицаТоваровДетальная_Строка`, `ТаблицаТоваровДетальная_Подвал` |
| 5 | `ТаблицаТоваровИтоги` | Только итоговая строка (общая сумма, НДС) | Выкл | Товары | `ТаблицаТоваровИтоги` |
| 6 | `ЭтапыОплаты` | График оплаты | Выкл | — | `ЭтапыОплаты_Шапка`, `ЭтапыОплаты_Строка` |
| 7 | `ПодписиПолные` | Руководитель + бухгалтер + М.П. | Вкл | Подписи | `ПодписиПолные` |
| 8 | `ПодписиМенеджер` | Только менеджер | Выкл | Подписи | `ПодписиМенеджер` |
| 9 | `ПримечаниеКДокументу` | Комментарий из заказа | Выкл | — | `ПримечаниеКДокументу` |

Блоки одной группы (Шапка: 1/2, Товары: 4/5, Подписи: 7/8) — взаимоисключающие. Выбирается один из вариантов.

**Соглашение по именованию:** идентификатор блока = идентификатор в коде конструктора. Области макета для составных блоков (таблица товаров, этапы) используют суффиксы `_Шапка`, `_Строка`, `_Подвал`. Логика `Макет.ПолучитьОбласть()` в процедуре формирования обращается к областям через соответствие «блок → массив областей».

### Создаваемые объекты метаданных

| Объект | Тип | Назначение |
|---|---|---|
| `ПечатьСчетНаОплатуКонструктор` | Обработка | Модуль менеджера: серверная логика формирования ПФ |
| `ПФ_MXL_СчетНаОплатуКонструктор` | Макет обработки | MXL-макет с именованными областями под каждый блок |
| `НастройкаБлоков` | Форма обработки | Интерфейс выбора блоков и порядка |
| `ПечатьСчетНаОплатуКонструкторКлиент` | Общий модуль (клиент) | Обработчик команды печати — открывает форму `НастройкаБлоков` |
| `НастройкиКонструктораПечати` | Регистр сведений | Хранение пресетов по пользователю |

### Структура регистра `НастройкиКонструктораПечати`

| Вид | Имя | Тип |
|---|---|---|
| Измерение | `Пользователь` | СправочникСсылка.Пользователи |
| Измерение | `ИмяПечатнойФормы` | Строка(150) |
| Ресурс | `НастройкиБлоков` | ХранилищеЗначения |

**Почему не типовой `НастройкиПечатиОбъектов`:** типовой регистр хранит настройки комплекта печати (какие формы печатать, количество копий), а не внутреннюю структуру блоков одной формы. Смешивание семантики нарушит типовую логику.

### Схема вызова

```
Документ.ЗаказКлиента
  → Печать → Меню «Счёт на оплату (конструктор)...»
    → КомандаПечати.Обработчик вызывает клиентскую процедуру
      → ОткрытьФорму("Обработка.ПечатьСчетНаОплатуКонструктор.Форма.НастройкаБлоков",
          параметр: МассивОбъектов — ссылки на выбранные ЗаказыКлиентов)
        → Пользователь выбирает блоки, задаёт порядок
        → Нажимает «Сформировать»
        → Сервер: СформироватьПечатнуюФорму(МассивОбъектов, НастройкиБлоков)
          → Получение данных запросом (аналогично типовому СчетНаОплату)
          → Для каждого блока из НастройкиБлоков:
              Макет.ПолучитьОбласть(ИмяБлока) → заполнить параметры → Вывести
        → Результат → УправлениеПечатью (типовой предпросмотр, печать, PDF)
```

### Регистрация печатной формы и механизм вызова

**Точка интеграции:** процедура `ДобавитьКомандыПечати()` в менеджере `Документ.ЗаказКлиента` (ManagerModule.bsl:3271). Именно здесь формируются все команды печати для этого документа, включая типовые счета на оплату.

`Документ.ЗаказКлиента` уже зарегистрирован в `УправлениеПечатьюУТСервер.ПриОпределенииОбъектовСКомандамиПечати` (Module.bsl:89), поэтому **отдельная регистрация в `ПриОпределенииОбъектовСКомандамиПечати` не нужна**.

**Добавляемый код** (в `Документ.ЗаказКлиента.ДобавитьКомандыПечати`, перед вызовом `ЗаказКлиентаЛокализация.ДобавитьКомандыПечати`):

```bsl
// Счёт на оплату (конструктор)
КомандаПечати = КомандыПечати.Добавить();
КомандаПечати.Обработчик    = "ПечатьСчетНаОплатуКонструкторКлиент.ОткрытьКонструктор";
КомандаПечати.МенеджерПечати = "";
КомандаПечати.Идентификатор = "СчетНаОплатуКонструктор";
КомандаПечати.Представление = НСтр("ru = 'Счёт на оплату (конструктор)...'");
КомандаПечати.ПроверкаПроведенияПередПечатью = Истина;
```

Свойство `Обработчик` указывает на клиентский общий модуль. Подсистема печати вызывает обработчик через `Вычислить()` (УправлениеПечатьюСлужебныйКлиент:128-129), поэтому обработчик **обязательно должен быть функцией** с сигнатурой:

```bsl
// Параметры:
//   ПараметрыПечати - Структура - см. УправлениеПечатьюКлиент.ОписаниеПараметровПечати:
//     * ОбъектыПечати - Массив - ссылки на печатаемые документы
//     * Идентификатор, МенеджерПечати, ДополнительныеПараметры и др.
//
Функция ОткрытьКонструктор(ПараметрыПечати) Экспорт
    ОткрытьФорму("Обработка.ПечатьСчетНаОплатуКонструктор.Форма.НастройкаБлоков",
        Новый Структура("МассивОбъектов", ПараметрыПечати.ОбъектыПечати));
    Возврат Неопределено;
КонецФункции
```

Паттерн аналогичен `ЗаданиеТорговомуПредставителю` (ManagerModule.bsl:231) и типовым командам `КомплектДокументов` (ManagerModule.bsl:3430).

**Дополнительный объект метаданных:**

| Объект | Тип | Назначение |
|---|---|---|
| `ПечатьСчетНаОплатуКонструкторКлиент` | Общий модуль (клиент) | Функция-обработчик `ОткрытьКонструктор(ПараметрыПечати)` — открывает форму `НастройкаБлоков` с передачей `ПараметрыПечати.ОбъектыПечати` из параметров команды. Обязательно функция, т.к. подсистема вызывает обработчик через `Вычислить()` (УправлениеПечатьюСлужебныйКлиент:128-129) |

### Ключевые типовые механизмы, которые задействуются
- `УправлениеПечатью.НужноПечататьМакет()` — проверка необходимости формирования
- `УправлениеПечатью.ВывестиТабличныйДокументВКоллекцию()` — вывод результата
- `УправлениеПечатью.МакетПечатнойФормы()` — получение макета
- `ФормированиеПечатныхФорм.ВывестиЛоготипВТабличныйДокумент()` — логотип организации
- `ШтрихкодированиеПечатныхФорм.ВывестиШтрихкодВТабличныйДокумент()` — штрихкод
- Типовой предпросмотр, печать и сохранение в PDF — из коробки БСП

## 3. Техническое обследование

### Существующие типовые объекты (не изменяются)

| Объект | Роль |
|---|---|
| `Обработка.ПечатьСчетовНаОплату` | Типовая печать счёта — остаётся без изменений |
| `Обработка.ПечатьЗаказовНаТоварыУслуги` | Типовая печать заказа — остаётся без изменений |
| `РС.НастройкиПечатиОбъектов` | Типовой регистр настроек комплекта — не трогаем |
| `ОМ.УправлениеПечатью` | Типовой API печати — используем, не изменяем |
| `ОМ.ФормированиеПечатныхФорм` | Утилиты формирования — используем |

### Минимальные изменения типовой

| Изменяемый типовой объект | Что именно меняется |
|---|---|
| `Документ.ЗаказКлиента.ManagerModule.bsl` | Добавление команды печати в `ДобавитьКомандыПечати()` (блок ~5 строк, перед вызовом `ЗаказКлиентаЛокализация.ДобавитьКомандыПечати`) |

Все остальные объекты — новые, типовые модули (`УправлениеПечатьюПереопределяемый`, `УправлениеПечатьюУТСервер` и др.) не изменяются.

### Потенциальные риски

| Риск | Вероятность | Митигация |
|---|---|---|
| Конфликт при обновлении `Документ.ЗаказКлиента.ManagerModule` | Средняя | Изменение локализовано (5 строк в конце `ДобавитьКомандыПечати`), легко перенести при обновлении |
| Некорректная разметка страниц при нестандартных комбинациях блоков | Низкая | Тестирование крайних случаев |
| Потеря настроек при смене пользователя | Низкая | Привязка к справочнику Пользователи |

### Что обязательно проверить вручную
1. Корректность MXL-макета — все области именованы, не пересекаются
2. Печать при различных комбинациях блоков (крайние случаи: только итоги, только шапка)
3. Сохранение в PDF через типовой механизм — разметка страниц при разных комбинациях
4. Права доступа менеджера к регистру настроек
5. Работа при первом использовании (пресет ещё не сохранён — подставляются значения по умолчанию)
6. Документы с разным составом данных (с/без НДС, с/без этапов оплаты, пустой комментарий)
7. Взаимоисключающие блоки — нельзя включить оба варианта одновременно
